<a id="项目概览"></a>
## 1. 项目概览

NTU Intercept 是一个基于 Three.js 的浏览器三维战斗沙盘。玩家可选择采用 Mk 10 双臂发射架的 USS Long Beach (CGN-9)，或采用 Mk 41 VLS 与 AEGIS/SPY-1B 的 USS Lake Champlain (CG-57)，面对 P-15 Termit、P-500、P-700、Kh-22 和 RGM-84 Harpoon 来袭武器并管理完整防空链路。

战斗重点不是简单的“目标进入圆圈后按概率消失”，而是让拦截结果由一条可观察的作战链路产生：

```text
雷达扫描 -> 带误差航迹 -> 3D 火控解算 -> 发射架转向
-> 导弹中段舰载数据链 -> 末段导引头或照射 -> 命中/脱靶
-> 漏防、电子对抗、子系统损伤 -> AAR 复盘
```

<a id="设计目标"></a>
### 设计目标

- 使用真实名称和时代技术关系，性能值保持明确的游戏化属性。
- 在三维空间中模拟速度、加速、阻力、能量、转弯率和有限射程。
- 让传感器误差、刷新率、航迹质量和火控延迟影响是否能够开火。
- 明确表现“舰载中段引导 + 末段导引头/照射”的两阶段制导。
- 通过通道、照射器、发射架机械周期和弹药形成饱和攻击压力。
- 让 ECM、箔条、烧穿距离和假目标捕获成为动态过程。
- 让漏防造成具体设备损伤，并继续改变后续战斗。
- 提供可移动镜头、战术雷达和完整 AAR 时间线。

<a id="当前实现范围"></a>
### 当前实现范围

当前版本是单舰防空与舰对舰交战沙盘。Slava 级 Moskva 既是具备 16 个实体 P-500 发射槽的敌方平台，也是可搜索、跟踪和毁伤的水面目标；CGN-9 与 CG-57 均可从实体 Mk 141 发射 RGM-84 Harpoon。项目仍不包含舰队级 CEC、航空兵、潜艇、多人联网或完整任务编辑器。舰船与武器模型为程序化生成，不依赖外部 3D 模型文件。

## 2. 快速开始

<a id="环境要求"></a>
### 环境要求

- Node.js 20.19+，或 Node.js 22.12+
- npm
- 支持 WebGL 2 的现代浏览器

<a id="安装与运行"></a>
### 安装与运行

```bash
npm install
npm run dev
```

打开 Vite 输出的地址，默认通常是：

```text
http://127.0.0.1:5173/
```

如果 5173 已被占用，Vite 会自动选择其他端口，以终端输出为准。

<a id="生产构建"></a>
### 生产构建

```bash
npm run build
```

构建命令依次执行 TypeScript 类型检查和 Vite 生产打包，输出位于 `dist/`。

## 3. 操作说明

<a id="战斗控制"></a>
### 战斗控制

| 控件 | 作用 |
|---|---|
| `AUTO FIRE` | 启用或暂停自动防空火力规划 |
| `DOCTRINE` | 在 SINGLE、DOUBLE、SS-L-S 条令间切换 |
| `RADAR` | 在主动辐射与 EMCON 静默间切换 |
| `SEARCH` | 在 360°、120°、60° 搜索宽度间切换 |
| `SLEW` | 将聚焦搜索轴指向当前选定航迹 |
| `CIWS` | 将近防系统设为 AUTO 或 HOLD |
| `THREAT CHAFF` | 控制来袭导弹是否投放自身箔条 |
| `OPFOR ECM` | 控制来袭武器对舰空导弹的干扰，以及敌方平台对 Harpoon 的持续 ECM 辐射 |
| `OPFOR DECOYS` | 独立控制敌方平台有限库存诱饵的自动抛放或 HOLD |
| `OPFOR RADAR` | 控制敌平台雷达辐射；静默会阻止新火控航迹并使在飞武器转入惯性外推 |
| `SHIP ECM` | 将舰载 ECM 设为 AUTO 或 HOLD |
| `SRBOC` | 将 Mk 36 箔条发射设为 AUTO 或 HOLD |
| `WEAPON` | 在 RIM-67、SM-2MR、SM-2ER 间切换 |
| `TARGET` | 循环选择仍然存活的目标 |
| `LAUNCH SAM` | 对当前目标执行人工发射请求 |
| `TIME` | 在 1X、2X、4X 仿真速度间切换 |
| `SCENARIO SETUP` | 暂停并打开场景编辑器 |

<a id="场景编辑器"></a>
### 场景编辑器

场景编辑器可配置：

- 来袭导弹型号、数量、发射间隔、初始高度、中心坐标和编队宽度。
- 攻击来源可选择传统空中编队，或从平台目录选择 Slava 级 Moskva；平台选择会按武器槽合同自动过滤主/第二波导弹类型和最大装填数。
- 齐射或逐发进入方式。
- USS Long Beach 的初始坐标。
- RIM-67、SM-2MR、SM-2ER 和 CIWS 弹药。
- SAM 交战通道与舰型相关的火控照射器数量。
- 第二波型号、数量和延迟。
- `SEA SKIMMER`、`SATURATION`、`HIGH SPEED` 快速预设。
- 在战术雷达上直接点选舰船或来袭编队位置。

场景重新开始时会清空航迹、AAR、爆炸、火灾、导弹、照射任务和子系统损伤，并复位两座 Mk 10 发射架。

<a id="仿真主循环与时间系统"></a>
## 4. 仿真主循环与时间系统

逻辑使用固定 `0.05 s` 仿真步长。显示帧率改变不会改变导弹动力学、机械周期或火控更新顺序；1X、2X 和 4X 只改变每个真实时间片累计多少仿真时间。

每个固定步长的大致顺序为：

```text
CIWS -> 助推器残骸 -> 箔条/诱饵
-> 敌方平台传感器/发射槽 -> 雷达与航迹更新 -> 舰船 OODA 机动 -> Mk 10 机械状态
-> 火力规划与照射器分配 -> 拦截弹制导/命中判断
-> 来袭导弹导引头、ECM、机动与舰体命中
-> AAR 快照
```

视觉雷达旋转、海面波浪、火焰、烟雾和部分 HUD 更新独立于固定步长，但不会直接决定战斗结果。

<a id="传感器与航迹"></a>
## 5. 传感器与航迹

<a id="雷达模型"></a>
### 机械扫描与 AN/SPY-1B 相控阵

| 雷达 | 维度 | 基础刷新 | 游戏最大距离 | 高度能力 | 主要作用 |
|---|---:|---:|---:|---|---|
| AN/SPS-48E | 3D | 0.75 s | 65 km | 有 | 高度估计与火控解算 |
| AN/SPS-49 | 2D | 1.15 s | 105 km | 无 | 远程预警与平面航迹 |
| AN/SPY-1B | 3D 相控阵 | 0.42 s | 82 km | 有 | 多功能搜索、跟踪与火控解算 |

60°/120° 聚焦搜索会缩短重访时间，并对测量质量提供加成。AN/SPS-48E 与 AN/SPS-49 属于机械扫描模型，把天线时间投入聚焦扇区后不会刷新扇区外目标。CG-57 的四面固定 AN/SPY-1B 使用独立的电子扫描模型：聚焦扇区获得快速刷新，同时以较慢周期维持 360°背景搜索，因此不会因为火控资源集中而整圈失明。SPS-49 可以先产生 2D 预警航迹，却不能单独满足舰空导弹发射所需的高度解算。

<a id="探测模型"></a>
### 探测概率与雷达地平线

有效探测距离按目标 RCS 的四次方根缩放：

```text
effectiveRange = maxRange * (max(0.05, targetRCS / 0.5))^0.25 * sensorHealth
```

系统同时计算雷达高度、目标高度、距离比、搜索聚焦和雷达地平线。通过距离门限后仍需进行确定性伪随机探测判定：远距离、低空、低 RCS、设备受损都降低单次扫描成功率。

这里的公式关系用于创造合理玩法，常数和距离均为游戏缩放值。

<a id="航迹模型"></a>
### 航迹关联、误差与火控解算

探测结果不是直接复制目标真值。每次测量带有位置和高度误差，误差随测量质量变化。航迹使用预测位置、速度、重访间隔和不确定度门限进行关联。

- 航迹质量随时间衰减，不确定度持续增长。
- 3D 高度数据超过约 4 秒未更新后失效。
- 航迹按质量分为 `unknown`、`suspect`、`classified`。
- 目标显著机动会重置或降低火控解算质量。
- 新鲜 3D 航迹会逐步积累 `solutionQuality`，达到 0.45 后才允许发射。
- 航迹年龄超过 2.2 秒时，发射请求被视为数据陈旧。
- 极低质量或超过 160 秒未更新的航迹会删除。

<a id="来袭武器模型"></a>
## 6. 来袭武器模型

| 型号 | 巡航高度 | 末段高度 | 巡航/末段速度 | 末段开始 | 游戏伤害 | 特征 |
|---|---:|---:|---:|---:|---:|---|
| P-15 Termit | 98 m | 12.5 m | 6.2 / 6.4 u/s | 24 km | 32% | 约 13 海里主动雷达开机，保持 320 英尺巡航至最后约 5.4 km 急降 |
| P-500 | 60 m | 15 m | 8.8 / 9.6 u/s | 18 km | 28% | 低空掠海、末段机动 |
| P-700 | 130 m | 20 m | 9.8 / 10.8 u/s | 22 km | 38% | 更高速度和更强末段摆动 |
| Kh-22 | 18,000 m | 110 m | 13.2 / 15.2 u/s | 45 km | 46% | 高空高速俯冲、CIWS 效率低 |
| RGM-84 Harpoon | 45 m | 6 m | 5.8 / 6.4 u/s | 13 km | 20% | 小型低 RCS 亚音速掠海弹，末段主动导引与快速小幅机动 |

来袭弹按 `inbound -> midcourse -> terminal` 转换状态。高度、速度和机动幅度在进入末段后连续插值，而不是瞬间切换。导弹具有有限转弯率、速度响应、滚转表现、主动导引头开机、视线目标和最近接距离记录。Harpoon 在约 13 km 进入末段，主动导引头开机并从约 45 m 连续下降至约 6 m；其末段蛇形机动频率较快但幅度显著小于 P-700，3D 模型同时显示贴海尾迹、导引头视场和随转弯变化的滚转。游戏化模型让多数 Harpoon 保持直接掠海，少数采用最后约 4.8 km 的跃升俯冲；强舰载干扰还可能暴露辐射源方位并触发简化的干扰源归向状态。

P-15 的导引头开机与降高使用两个独立门限：约 24 km 进入主动雷达制导，但仍保持约 320 英尺巡航；只有最后约 5.4 km 才快速下降。参考包线图给出约 21.6 海里总航程，并显示导弹在 18.7 海里位置仍维持 320 英尺，因此剩余约 2.9 海里（5.4 km）是独立末段降高区，而不是把它误做成全程贴海弹。

### 敌方发射平台闭环

Slava 级 Moskva 是首个目录化敌方平台。其定义声明 MR-800、MR-700 与 Argument 传感器槽，以及一个兼容 P-500 的 `inclined-canister` 武器槽；模型则通过标准 `platformSlots` 清单提供 16 个实体 Bazalt 硬点、各自管口方向和可释放端盖。运行时首先验证“定义容量 = 模型硬点数”，再为主波和第二波预约未占用硬点，并以跨波次共享的 `nextAvailable` 保持机械发射间隔。

平台发射的 P-500 不会直接生成在空中。导弹从具体管口世界坐标出现，依次经历 `TUBE EXIT -> BOOST -> PROGRAM TURN -> MIDCOURSE TAKEOVER`，离舰段使用槽位声明的管轴、出口速度、助推持续时间与接管时间；接管后才进入通用 P-500 包线、雷达高度和末段制导逻辑。端盖状态与硬点状态保持一一对应，已经发射的硬点不能被第二波再次预约。

Moskva 的传感器维护一条独立于真值的目标航迹，包含位置/速度误差、质量、不确定度和刷新时间。每次扫描还按目标 RCS 四次方根缩放有效距离，计算双方天线高度的雷达地平线，并通过确定性探测概率判定是否真的获得新测量；漏检只衰减传感器质量并保留新鲜航迹记忆，不会伪造一条新报告。P-500 中段按武器槽声明的 1.4 秒数据链周期与 0.35 秒链路延迟接收指令点；刷新间隔内按估计速度外推。平台雷达静默、航迹质量不足或平台被毁后，导弹记录 `PLATFORM DATALINK LOST` 并沿最后估计惯性飞行，而不是继续读取己方舰船真值。

中段导航点进入预定末段区后，P-500 先进入 `ACTIVE SEEKER SEARCH`。只有真实目标同时处于游戏化捕获距离与 50° 视场内才会记录 `TARGET ACQUIRED`，随后停止平台数据链并进入 `terminal-autonomous`；舰载 ECM、SRBOC 箔条、HOJ 和末段机动只在主动导引头完成捕获后参与解算。前端状态链为 `SHIP GUIDED -> INERTIAL`（丢链时）`-> ACTIVE SEARCH -> ACTIVE`。

平台武器槽还声明最低航迹质量、最小连续航迹龄与火控反应时间。预约并不等于立即发射：Moskva 必须先连续保持 2.4 秒合格航迹，再等待 1.6 秒火控/指挥延迟，才能逐管放出 P-500。若多枚弹在等待期间同时到期，运行时仍按场景间隔与发射器最小间隔执行实际释放，不会在火控就绪帧同时喷出。平台被击毁后，尚未离筒的预约弹转为独立的 `canceled` 状态并保留端盖，已经离舰的导弹继续完成攻击，因此 `fired` 计数不会混入取消弹。

### 舰对舰交战闭环

两艘可选舰均声明数据驱动的 `surfaceStrike` 能力与 8 个 Mk 141 实体硬点。己方雷达为敌平台维护一套独立水面航迹，受刷新率、误差、RCS、雷达状态和航迹质量影响；关联 ID 变化、质量跌破门槛或报告龄超过 4 秒都会中断连续确认。过期航迹仍可作为态势历史显示，但不能继续建立火控、发射新弹或刷新在途弹数据链。CG-57 需要 2.2 秒连续确认加 1.6 秒火控延迟，CGN-9 需要 2.8 秒加 2.2 秒。火控规划根据情报中的目标耐久、武器毁伤、先验漏防率以及已解析武器的命中/拦截结果计算每波规模：首波最多 4 枚，上一波仍在途或处于 BDA 等待窗时禁止追加，重攻则按估算剩余所需命中数缩减。`SURFACE STRIKE` 保持开关和 `LAUNCH HARPOON` 手动控制也服从相同门控，不能绕过在途武器上限。

平台场景会把原有 AIR RAID 状态卡切换成 `SFC` 水面接触卡。航迹质量不足时只显示 `SURFACE CONTACT`；完成分类后才显示舰名。卡片用 `TRACK BUILD -> FC ASSIGN -> BDA PENDING` 表示 OODA 进度，并持续显示空中威胁数、表面距离、航迹质量、OPFOR EW 状态、Harpoon 剩余量和区间化 BDA。水面雷达还计算双方天线高度与显著目标高度的地平线：越过直接视距时只保留低概率、低质量的间歇接触，卡片标记 `HORIZON LIMITED`，事件流记录间歇接触，不能建立火控；进入视距后标记 `LOS` 并恢复正常测量。命中后先显示舰型定义的 `BDA ASSESS` 等待窗，随后根据己方目标耐久情报、已确认命中数、武器效果误差和当前航迹质量给出剩余战力区间或失能置信度；战斗 HUD 与日志不再打印敌舰真实剩余耐久和真实受损子系统。

Harpoon 从具体发射箱轴线离舰，经历 `boost -> datalink midcourse -> inertial coast（失链时）-> active search -> terminal autonomous`。中段数据链每次只传递带误差的位置与速度估计，刷新间隔内由弹上状态外推；己方雷达静默、航迹质量不足或丢轨后，导弹沿最后指令点惯性飞行。导航点进入 13 km 游戏化末段区只会启动主动搜索，真实目标还必须进入 50° 视场和捕获距离才会记录 `TARGET ACQUIRED`。只有捕获后才能追踪平台真值、加入有限速度提前量并参与敌舰 ECM/诱饵解算；未捕获的导弹会越过最后指令点并独立判定脱靶。

敌平台点防御不再直接读取鱼叉真值距离。Moskva 为每枚来袭弹维护独立航迹：在 11.5 km 游戏化探测包线内按 0.72 秒刷新，掠海高度与雷达子系统损伤降低发现概率和航迹质量；航迹低于 `0.30` 或超过 3.2 秒未刷新即失效。合格航迹还需经过 1.35 秒火控反应，之后才能占用两条独立点防御通道之一。平台根据观测航迹的闭合速度、估算命中倒计时、距离、航迹质量和 4.5 km 局部航迹密度形成确定性威胁排序；beam 机动与点防御目标分配共享这一顺序，而不是按鱼叉创建顺序开火。命中概率同时受航迹质量、位置不确定度、系统健康度和已探测局部饱和度影响，未被雷达发现的弹不会凭真值产生饱和惩罚。近防健康度还同时改变有效通道、火控反应和通道循环：Moskva 在 45% 健康时由两通道降为一通道，反应倍率为 1.44、循环倍率为 1.55；5% 以下完全停机但不会删除航迹。场景面板的 `OPFOR POINT DEFENSE HEALTH` 可直接验证这些状态。射击不会在同一帧判定结果：游戏化综合近防效应器按 4 km/s 飞向射击时的解算点，通道在效应器到达前保持占用，来袭弹继续运动；若鱼叉先命中，迟到的射击不会倒推成拦截。Moskva 另有 6 个游戏化“有效射击批次”，代表满足跟踪、稳定射击与冷却条件的完整拦截窗口，而非现实炮弹颗数；每次开火立即消耗一个，在途效应器仍可结算，库存耗尽后的合格航迹会记录一次 `MAGAZINE DEPLETED` 并继续突防。近防采用有限的“射击－观察－再交战”：第一发脱靶会保留来袭弹和航迹；结算后同一航迹还必须单独等待 0.9 秒观察与重新解算，不能立即转交另一条空闲通道。Moskva 每个目标最多两次，第二次仍失败则形成漏防。`OPFOR RADAR: SILENT` 会阻止新防御航迹形成，但 ECM 仍由独立开关控制。

目标被摧毁不会批量删除其他在途 Harpoon。每枚弹独立记录目标丢失时刻，停止数据链与真值归向，继续飞向最后已知点，随后以 `TARGET DESTROYED / AIMPOINT LOST` 结束；因此 AAR 和三维画面都能保留齐射中每一枚弹的生命周期。

Harpoon 命中不再使用“距离舰体中心 7.5 单位”的球形判定。导弹位置先转换到目标平台局部坐标，再与按真实模型长宽生成、舰艏舰艉收窄的碰撞轮廓比较；平台 beam 机动因此会自然形成舷侧命中。接触只记录 `PENETRATION` 并进入不可再拦截的穿透阶段，此时舰体与子系统不扣血；RGM-84 配置的 0.32 秒游戏化延时引信到期后才记录 `INTERNAL DETONATION`、生成舰内爆炸并开始 BDA。Moskva 的 BOW/FORWARD/AMIDSHIPS/AFT 分区把实际局部纵向命中位置映射到雷达、武器、动力、近防或电子战系统候选，取代全舰随机选系统。

每次舰内爆炸还会按命中水线高度与舷侧深度生成独立 casualty，分别记录火灾和进水强度。平台每 2 秒进行一次游戏化损管结算：`damage-control` 健康度在受控衰减和失控增长系数之间插值，每个周期造成渐进舰体损失；强度低于阈值后记录 `CONTAINED`，否则可能在导弹全部落地后继续恶化并独立造成 `DISABLED BY PROGRESSIVE DAMAGE`。场景面板的 `OPFOR DAMAGE CONTROL HEALTH` 可直接设置能力；只要幸存平台仍有 casualty，任务会保持 `DAMAGE ASSESSMENT` 而不会提前弹出 AAR。AAR 的 `PROG DAMAGE` 单独汇总这部分损失。

Moskva 的最大/巡逻/巡航航速、加速度、转弯率、决策周期和防区距离环全部来自平台定义。它每秒执行一次平台 OODA：无有效目标航迹时巡逻；仅依据带误差的目标航迹选择接近、保持距离或撤离；出现合格的 Harpoon 来袭航迹时，选择转向量较小的垂直威胁轴方向并加速到可用最大航速。目标真值只用于生成雷达或 ESM 测量，不直接参与机动决策。推进与舰体毁伤会限制实际速度和转向，平台被摧毁后进入 `disabled` 并逐步停车。命中还会降低随机子系统健康度并留下持续火焰与烟柱；雷达毁伤会降低航迹质量并减慢旋转，点防御和电子战损伤分别降低硬杀伤与软杀伤能力。任务结算会等待来袭弹、Harpoon、发射队列和舰载发射器全部结束，再区分水面目标被摧毁或反舰弹耗尽后目标幸存。

平台对平台场景不再因雷达地平线形成永久死锁。对方雷达开机时，舰载电子支援系统会生成 `0.12-0.20` 质量、带大距离误差的被动辐射源航迹；该航迹以紫色虚线和 `ESM` 菱形显示，只允许 OODA 选择接近，不能累计武器火控航迹龄。双方进入有机雷达地平线并取得合格直接航迹后，才开始 Harpoon 或 P-500 发射。选择舰载攻击平台时沙盒默认约 42 km 舰间距，便于立即观察交战；手动拉到 60 km 以上则会先经历 ESM 引导接近。

Harpoon 齐射按舰型参数分配左右交替的中段航路偏置、汇聚距离、共同计划到达窗和有限速度补偿。导弹不会瞬移或突破配置的速度修正上限；偏置在主动导引头开机前逐步消除。运行 `npm run verify:surface-salvo` 会同时验证近距双方离舰、四枚 Harpoon 的相反航路和到达时间，以及远距 ESM 接近但不开火的约束。

水面接触卡会同步显示敌方交战链状态：`OPFOR ESM CUE -> TRACK BUILD -> FC BUILD -> FC READY -> LAUNCHED`，因此等待雷达地平线、连续航迹龄或火控延迟时不再表现为无反馈。`npm run verify:bilateral-launch` 分别让 CGN-9 和 CG-57 对 Moskva 作战，要求两艘美舰各自发射 Harpoon、Moskva 对两种目标均完成 P-500 实体离筒，并在桌面与移动视口检查状态卡布局。

### 两阶段制导

发射时导弹从实际 Mk 10 导轨位置和方向离架。中段并不直接追踪目标真值，而是使用火控航迹生成的 `commandPoint` 和 `commandVelocity`：

1. 舰载系统按航迹质量产生带延迟的数据链更新。
2. 更新间隔随航迹质量变化，数据丢失时导弹按惯性外推。
3. 弹道按目标剖面选择：P-15、P-500、P-700 与 Harpoon 使用低空前置拦截走廊；Kh-22 等高空目标才采用 loft 弹道并在末段下降接敌。Mk 41 对低空目标只保持短暂的垂直安全离舰段，随后快速执行程序转弯。
4. 进入武器末段范围后，转交主动导引头或当前舰型的火控雷达照射。

<a id="rim67末制导"></a>
### RIM-67 主动末制导

这里的 RIM-67 主动导引头是为“两阶段制导”玩法设计的游戏化处理，不代表现实型号的严格配置。

- 进入末段后导引头开机，并有约 0.35 秒预热。
- 捕获概率考虑视场角、距离、交接误差、目标 RCS、邻近竞争目标和海杂波。
- 捕获后以有限刷新率生成带噪声的导引头瞄准点。
- 视轴角或视线角速度超过万向架能力时会断锁。
- 断锁后短时间使用航迹记忆；长期无法捕获则判定脱靶。

<a id="sm2照射"></a>
### SM-2 与舰载照射器

SM-2MR/ER 在中段同样接受舰载更新，进入末段后需要舰载火控雷达连续照射。USS Long Beach 使用 2 部 AN/SPG-55；USS Lake Champlain 使用 4 部 AN/SPG-62。照射能力由当前选择舰船的实际模型挂点和设备数量决定。

- 照射器必须先转向目标，进入角度门限后才能建立照射。
- 照射通道有限，同一目标可以共享已建立的照射。
- 方位解算使用舰体本地坐标，舰船转向不会改变照射器对目标方位的判断。
- 超过约 2.5 秒没有照射，SM-2 会因照射丢失而脱靶。
- 火控子系统损伤会降低转向速度，并按所选舰船的实际照射器数量减少可用通道。

<a id="拦截动力学"></a>
### 飞行动力学与命中判定

每个拦截弹保留速度矢量、飞行距离、能量、助推状态和轨迹历史。更新过程包含：

- 助推段加速与助推器分离。
- 最大转弯率限制和方向渐进插值。
- 大转弯需求引起的阻力与能量损失。
- 中段抬高、末段机动和最大射程耗尽。
- 线段最近点计算，避免高速目标在单帧内穿过命中球。
- 由制导质量、几何关系、剩余能量、饱和度和照射状态共同形成最终 PK。
- 未命中后使用最近接距离和距离再次增加判断是否已经飞越目标。

### 通道与目标分配

- 发射架已接受但尚未离架的任务也占用 SAM 通道和目标分配数。

- 自动规划器只选择具备新鲜 3D 火控解算且处于武器射程内的目标。

<a id="mk10发射架"></a>
## 9. Mk 10 双臂发射架

前、后两座 Mk 10 都使用固定步长机械状态机：

```text
READY -> SLEWING -> FIRING -> RETURNING -> LOADING -> READY
```

1. 火控分配任务后，发射架使用当前雷达航迹计算相对方位与俯仰。
2. 方位约以 55°/s、俯仰约以 25°/s 的游戏基础速率转动。
3. 进入约 2° 对准门限后，从选定导轨模型取得真实世界坐标和四元数。
4. 拦截弹以导轨方向作为初始速度和姿态，而不是从舰体中心生成。
5. 发射架短暂保持发射姿态，然后返回装填方位并降到零俯仰。
6. 已发射导轨的可见导弹从输弹位置移动回导轨，完成后交替使用另一臂。
7. 转架期间目标被摧毁或航迹长期丢失时，任务取消、弹药退回并复位。
8. 发射架损伤会减慢转向和装填；健康度低于 5% 后停止接受任务。

<a id="mk41序列器"></a>
### Mk 41 垂直发射序列器

CG-57 的前、后 Mk 41 阵列是两个可以并行工作的发射资源，但每个阵列内部都受独立点火序列器约束：

- 122 个实体单元分别绑定 `SM-2MR`、`SM-2ER` 或其他保留载荷；任务只能选择装有请求弹种的单元。
- 默认防空装载为 48 枚 SM-2MR、32 枚 SM-2ER 和 42 个其他载荷单元。沙盒输入超过 122 时按输入比例缩放，不能生成实体单元以外的虚拟弹药。
- 单元依次执行开盖、热发射、喷焰消散和关盖，发射后永久标记为已用。
- 开盖阶段即预留弹药；目标被摧毁或阵列故障导致的未发射取消会退还同一弹种，关盖后恢复 `ready`。只有真正离舰才进入 `spent`。
- 同一阵列的连续点火保留游戏化最小间隔；阵列健康度降低会延长该间隔。
- 正在开盖或热发射单元周围一格形成临时安全区，不能同时接受任务。
- 上一发的排焰仍在甲板附近时，相邻单元被硬性禁止；分配器优先选择距离上一发更远的可用单元。
- 前后阵列交替接受任务；一侧受损、忙碌或无安全单元时可转移到另一侧。
- 运行诊断记录两阵列的实测最小点火间隔、发射单元历史、各弹种 ready/pending 数量、已用单元和取消关盖数，用于检查齐射安全与弹药守恒。
- 阵列健康度低于 75% 后会按命中中心隔离局部单元簇；其中的防空弹成为被困弹药并从可发射数量扣除。健康度低于 5% 时，该阵列所有剩余未发射单元失效，另一阵列仍可独立工作。
- 场景编辑器可分别设置前、后发射系统初始健康度，用于复现部分损伤、单阵列失效和非对称弹药可用性。

<a id="电子战"></a>
## 10. 电子战与诱饵

<a id="威胁电子战"></a>
### 威胁侧 ECM 与箔条

`OPFOR ECM` 对 SM-2 末段瞄准点施加距离相关的干扰偏差。拦截弹接近目标后进入烧穿区，干扰效果下降并记录 `ECM BURN-THROUGH`。在水面交战中，ECM 表示敌舰电子战天线的持续辐射，并不是发射物；`OPFOR DECOYS` 独立控制实际离舰的有限库存诱饵。Moskva 配有 8 组游戏化诱饵，只有合格来袭航迹进入 9 km 观测部署区后才会按 2.2 秒基础冷却向航迹估计的舷侧释放橙色诱饵云。`electronic-warfare` 健康度控制干扰强度，独立的 `countermeasures` 健康度控制诱饵发射能力和冷却；场景面板分别提供 `OPFOR ECM HEALTH` 与 `OPFOR DECOY LAUNCHER HEALTH`。平台不会读取鱼叉真实坐标决定抛放方向或距离；只有鱼叉自身主动导引头捕获后，弹上软杀伤解算才使用自身测距。

鱼叉软杀伤解算比较敌舰回波、诱饵 RCS、弹到两者的距离、ECM 强度、电子战子系统健康度、HOJ 门槛与 2.4 km 烧穿距离。结果可能是 `DECOY REJECTED`、`ECM CONTESTED`、`ACTIVE / HOJ`、`BURN THROUGH`，或明确记录为 `DECOY`、`ECM`、`ECM + DECOY` 的软杀伤。纯诱饵捕获不再依赖 ECM 健康度；关闭 `OPFOR ECM` 只停止辐射，关闭 `OPFOR DECOYS` 只停止抛放，点防御仍独立工作。

来袭导弹还可投放自身箔条。SM-2 末段会比较目标与箔条的雷达截面积，可能发生 `DECOY CAPTURE`；箔条随时间扩散、漂移并衰减。

<a id="舰载ecm"></a>
### 舰载 AN/SLQ-32 ECM

舰载 ECM 不是发射出去的弹药，而是舰上电子战天线的电磁辐射。模拟使用信号/干扰比和导弹型号的游戏化烧穿距离：

- 远距离时 ECM 对导引头瞄准点形成动态偏移。
- 越接近舰船，目标回波相对增强，最终发生烧穿。
- AN/SLQ-32 健康度同时影响干扰功率和烧穿时机。
- ECM 本身通常不能稳定制造软杀伤，主要与 SRBOC 箔条协同工作。

<a id="srboc"></a>
### Mk 36 SRBOC

来袭弹进入末段后，SRBOC 自动向威胁轴线侧方发射箔条弹：

1. 弹丸从舰舷发射点沿二次贝塞尔弹道飞行。
2. 到达空爆点后生成具有初始 RCS、漂移速度和 14 秒寿命的箔条云。
3. 导引头比较舰体与箔条的距离四次方功率关系。
4. ECM 会提高锁定转移概率；进入烧穿区后假目标锁定会解除。
5. 导弹错过舰船并拉开到门限外时形成 `SOFT KILL`。

SRBOC 损伤会延长发射冷却与弹丸飞行时间，低于 5% 后无法发射。

<a id="机动与近防"></a>
## 11. 舰船机动与 CIWS

舰船按舰型目录声明的决策周期执行 OODA 机动。无有效水面航迹时以巡逻航速保持航向；获得水面航迹后，仅依据带误差的航迹位置选择接近、距离环保持或撤离，并使用巡航航速。存在有效近距来袭弹航迹时，防御优先级抢占水面机动：舰船选择转向量较小、与威胁轴线近似垂直的方向进行 beam 机动，并提升到受推进与舰体健康限制的最大航速。这会增加来袭弹末段转向负担。CGN-9 与 CG-57 的巡逻/巡航/最大航速、加减速、转弯率和距离环均来自各自配置，而非主循环硬编码。

敌平台中尚未离开发射筒的预约武器不会进入己方雷达测量或 `AIR ACTIVE` 计数。只有实体硬点完成释放后，导弹才能被雷达发现、形成空中航迹并触发 `defensive-beam`；因此敌雷达静默时，己方不会对“筒内 P-500 真值”提前规避。

推进系统和舰体完整度共同限制最高速度；推进损伤也降低加速度与转向率。

CIWS 只处理非常接近的存活目标：

- 检查前/后炮位射界、目标闭合速度、剩余窗口和转向误差。
- 目标在盲区、正在远离或窗口已经关闭时保持射击禁止。
- 每次射击消耗 60 发弹药。
- PK 受导弹型号、局部饱和数量、可用射击轮数和 CIWS 健康度影响。
- Kh-22 的高速末段对 CIWS 有严格 PK 上限。

<a id="子系统损伤"></a>
## 12. 子系统损伤

<a id="损伤分配"></a>
### 命中区段与损伤分配

漏防命中首先扣除舰体完整度，然后根据以下因素决定设备损伤：

- 导弹初始来袭方向相对舰体纵轴形成前部/中部/后部偏置。
- 每枚导弹固有的瞄准偏差。
- 小范围确定性破片扰动。
- 对应区段的候选设备池。

每次命中产生一个主损伤和一个较小的破片次损伤。事件、命中区段、设备名称和健康度变化都会进入战斗日志与 AAR。场景中的随机结果由固定种子产生，便于重放相同配置。

<a id="损伤影响"></a>
### 各子系统的实际影响

| 子系统 | 损伤后的实际效果 |
|---|---|
| AN/SPS-48E / AN/SPY-1B | 缩短 3D 探测距离、降低测量质量、延长刷新；失效后无法更新高度火控数据 |
| AN/SPS-49 | 缩短 2D 预警距离、降低质量、延长刷新 |
| AN/SPG-55 / AN/SPG-62 | 降低照射器转向速度和可用通道数量 |
| Mk 10 AFT/FWD | 独立降低对应发射架转向与装填速度；失效时取消任务并停用 |
| Mk 41 AFT/FWD | 延长对应阵列点火间隔并隔离局部单元；失效后该阵列剩余导弹成为被困弹药 |
| CIWS | 降低转向速度、射击频率和 PK；失效后停止跟踪与射击 |
| AN/SLQ-32 | 降低 ECM 功率，使敌导引头更早烧穿 |
| Mk 36 SRBOC | 延长冷却和弹丸飞行时间；失效后不能投放舰载箔条 |
| PROPULSION | 降低最高航速、加速度和转向率 |

`DAMAGE CONTROL` 面板显示 9 个设备的连续健康度。健康度低于 65% 显示降级，低于或等于 5% 显示 `FAIL`。

<a id="三维表现"></a>
## 13. 三维表现与舰船模型

场景设置中可以切换两艘程序化舰船。各舰拥有独立的纵向站位表，每站使用 8 点横剖面控制甲板缘、上舷折角、水线、龙骨与首部舷弧；共享模块只负责三角化，不会把一种舰体线型套到另一舰级。CG-57 目前使用 13 个纵向站位，并由实舰约 `172.8 m × 16.8 m` 和 `2.25 m/世界单位` 直接推导模型尺寸：`76.8 × 7.47` 世界单位，长宽比 `10.29:1`。加密的艉部、平行舯体、艏肩和飞剪艏站位改善了方艉收束、舷侧折线和艏部外飘。模型还包括前后 Mk 41 阵列、四面 SPY-1 阵面、双上层建筑、双桅/排烟组、机库、飞行甲板与前后 Mk 45 炮。Mk 41 使用独立舱盖和一次性单元状态，发射过程为开盖、热发射垂直爬升、延迟程序转弯、关盖并标记已用单元。

CG-57 模型配置 4 部 AN/SPG-62，前、后上层建筑两舷各一部；每部拥有独立转向、目标任务与末段照射波束。两座 Phalanx CIWS 也按各自前后射界选择目标，并在舰艇机动时使用舰体本地坐标保持正确跟踪。

CG-57 的纵向校准同步作用于舰体、一级设备、子系统命中位置、损伤分区和前后 VLS 发射原点，不对圆形火炮座、雷达阵面或桅杆截面做破坏性的非等比缩放。Mk 41、Mk 45、Phalanx、AN/SPG-62、AN/SPY-1 与 AN/SLQ-32 已提取为可复用的美国海军设备组件；舰载艇、救生筏筒、锚链孔和栏杆也由 CGN-9 与 CG-57 共用附件构造器生成，舰级文件只负责真实布局、数量和朝向。模型另有桥翼、Mk 141 四联装 Harpoon 发射箱、艇库凹槽与吊艇架、艏部防浪板、系泊柱、舷侧板缝和直升机甲板 RAST 引导线。

修改 CG-57 模型后，先启动开发服务器，再运行 `npm run verify:cg57`。脚本会选择 USS Lake Champlain、启动并暂停沙盒、生成斜前和正舷侧截图，同时把浏览器 console/page error 作为失败条件。当前截图验证的模型长宽比为 `10.29:1`，新防浪板位于前 Mk 41 前缘，未与舱盖或 Mk 45 穿模。

舰船选择器由 `SHIP_CATALOG` 自动生成；各舰定义集中提供模型构造器、传感器、弹药、发射系统与子系统位置。烟囱和 CIWS 等动态效果读取模型挂点，后续同类舰不需要在界面或射击代码中增加舰名判断。

舰船是针对 USS Long Beach 外形特征程序化生成的模型，包括：

- 按 CGN-9 比例收窄梁宽，保留方艉，并使用外飘上舷、折角、水线收束、龙骨曲线与升高舰艏构成多层截面。
- 舰桥、桅杆、甲板设备、舷号、国旗、航行灯和救生设备。
- AN/SPS-48E、AN/SPS-49、SPG-55 指示器与旋转动画。
- 前后 Mk 10 双臂发射架、耳轴、摇架、液压机构、装填导轨和安全标线。
- 前后 CIWS、Mk 36/电子战天线的视觉抽象。
- 航迹线、导引头视场、照射波束、尾焰、助推器分离和残骸。
- 海面、舰艉航迹、爆炸、持续火灾、烟雾和电子战脉冲。
- 舰体与导弹使用程序化粗糙度/法线贴图增强 PBR 表面细节；来袭弹尾焰和烟雾使用固定粒子池、`THREE.Points` 与自定义 Shader，避免逐帧创建 Mesh。
- WebGL 画面经过 `RenderPass -> SSAO -> UnrealBloom -> OutputPass`，并使用 ACES Filmic ToneMapping；宽度不超过 720 px 时关闭 SSAO，控制移动端 GPU 压力。
- 海面通过 `OceanSurface` 后端合同接入；当前为保持兼容的 `webgl-cpu-waves`，未来 WebGPU FFT 实现可替换后端而不进入战斗主循环。
- 近、中、远三档 LOD，动态导航灯与甲板灯，以及随雷达状态和健康度变化的 SPS-48 搜索扇面。

<a id="数值与单位"></a>
## 15. 数值、单位与确定性

- 水平比例：`10 world units = 1 km`。
- 高度输入：`1 world unit = 50 m`。
- 速度以世界单位/秒保存，不应直接解释为公开装备速度。
- 雷达距离、烧穿范围、转弯率、弹药和伤害均为游戏平衡参数。
- 概率事件使用确定性伪随机序列或确定性数学滚点；相同场景配置通常可复现相同结果。
- Three.js 使用 Y 轴为高度，舰船局部 +X 为舰艏方向。

<a id="模块边界"></a>
### 模块边界

当前运行时采用“能力配置 + 模型挂点 + 通用编排”结构。`main.ts` 不判断 `long-beach` 或 `ticonderoga` 等舰型 ID，也不包含 USS Long Beach 的几何、设备名称、默认库存或命中分区。具体舰型在目录中声明传感器、发射器、兼容武器、平台机动、雷达反射特征、弹药、子系统位置和损伤区段；模型文件只负责构造 Three.js 对象及设备挂点。

通用子系统使用 `primaryRadar`、`fireControl`、`forwardLauncher` 等语义槽位，而不是把 `SPS-48`、`SPG-55` 或 `Mk 10` 当作跨舰型 ID。机械扫描与固定阵面差异由传感器能力决定，Mk 10 与 Mk 41 的差异由发射器判别配置决定。新增舰型不应在 `main.ts` 中增加舰名分支。

<a id="扩展舰型与武器"></a>
### 扩展舰型与武器

新增舰型：

1. 在 `src/models/` 中实现模型构造器，并提供雷达、火控、发射器、CIWS、LOD 和灯光所需挂点。
2. 在 `src/ship-catalog.ts` 注册一个 `ShipDefinition`，填写平台性能、传感器、发射器能力、库存、语义子系统位置与舰体损伤分区。
3. 固定阵面舰只填写 `fixedSensorFaces`；机械扫描舰不需要伪造阵面配置。
4. 通过沙盒切舰，验证武器菜单、默认库存、雷达名称、发射架和损伤行为自动跟随配置。

新增来袭弹时，在 `src/threats/` 新建一个型号文件，把包线、RCS、CIWS 修正、末段/EW 能力、沙盒预设和程序化模型集中为一个 `ThreatDefinition`，然后只在 `catalog.ts` 注册一次。`EnemyType`、主/第二波菜单、预设按钮和参数查询都会自动生成。P-15 的独立降高门限、Harpoon 的 pop-up 与 HOJ 都是能力字段；出现新行为时应扩展通用能力合同，不应在 `main.ts` 增加型号判断。

新增敌方发射平台时，在 `src/platforms/models/` 构造模型并通过 `createPlatformModelSlots` 注册传感器锚点和武器硬点，然后在平台目录注册一个 `EnemyPlatformDefinition`。定义中的每个 `weaponSlot` 必须声明发射器族、兼容威胁、容量、最小间隔、出口速度、助推时间与制导接管时间；运行时会拒绝容量、槽 ID 或传感器锚点不一致的模型。不要在 `main.ts` 判断平台 ID。

新增拦截弹时，扩展 `WeaponType` 并在 `src/interceptor-data.ts` 添加飞行剖面，同时在舰型定义中声明发射器兼容性和库存。VLS 算法应进入 `src/vls.ts`，固定阵面算法应进入 `src/sensor-faces.ts`。更完整的约束见 [ARCHITECTURE.md](ARCHITECTURE.md)。

## 17. 开发与验证

基础质量门：

```bash
npm run build
```

建议每次修改战斗逻辑后至少验证：

1. 单目标人工/自动发射与完整 Mk 10 机械循环。
2. 多目标齐射下的通道和目标分配。
3. RIM-67 主动末制导捕获/断锁。
4. SM-2 的 SPG-55 照射与照射丢失。
5. ECM 烧穿、SRBOC 空爆、假目标锁定与软杀伤。
6. 漏防后的主/次子系统损伤和后续能力下降。
7. AAR 时间线、拖动、播放和事件跳转。
8. 桌面视口与 `390 x 844` 移动视口无溢出、无 UI 冲突。
9. 浏览器控制台没有运行错误。

仓库包含开发过程中的验证截图，可用于比较舰体、雷达、发射架、电子战、损伤和移动端布局。

<a id="已知边界"></a>
## 18. 已知边界与后续方向

- 当前可选择 USS Long Beach 与 USS Lake Champlain，但每个场景仍只有一艘防御舰，没有护航编队、预警机和 CEC 网络。
- Moskva 当前是可视化、带传感器/发射槽和真实离舰序列的攻击来源，但还不是可被搜索、分配反舰火力和承受损伤的完整水面战斗实体。
- 地球曲率、气象、海况和雷达传播采用简化关系，不是专业电磁模型。
- 没有连续六自由度刚体气动；转弯和能量模型是三维点质量近似。
- 导引头和 ECM 是可解释的概率/信号关系模型，不是射频工程仿真。
- 程序化舰船模型侧重可识别轮廓和作战设备，不是测绘级数字孪生。
- 损管目前没有维修班、火灾蔓延、供电网络和冗余线路重构。
- 尚未提供完整自动化测试套件；当前以 TypeScript 构建、机制场景脚本和 Playwright 浏览器场景验证为主。CG-57 模型可用 `npm run verify:cg57` 重复验证。
- 当前海面仍由 CPU 更新网格。计划中的 WebGPU FFT 后端需要频谱生成、水平/垂直逆 FFT、位移/法线纹理、Jacobian 泡沫计算，以及与 WebGPU 后处理节点兼容的渲染链；WebGL 将继续作为回退路径。
- 来袭弹烟雾目前是每枚导弹本地坐标中的固定粒子池，转弯时会随弹体弯折；长时间世界空间烟迹需要独立的场景级粒子历史池。

合理的后续方向包括舰队协同航迹、CEC 网络税、更多平台、维修/供电网络、任务存档和更细致的 AAR 数据导出。
